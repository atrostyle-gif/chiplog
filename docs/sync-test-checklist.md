# CHIPLOG ログ同期・画像同期 テスト観点チェックリスト

実装済み機能の「壊れないか」「表示が矛盾しないか」を体系的に確認するためのチェックリストです。

---

## 優先度の定義

| 優先度 | 意味 |
|--------|------|
| **P0** | 壊れると実運用に大きく影響する最重要ケース。例: 保存できない、同期できない、削除が壊れる、二重送信でデータ破損、正本不整合。 |
| **P1** | 実務上かなり重要だが、P0 ほど致命ではないケース。例: 画面復帰同期、一覧反映、状態表示、非同期復帰時の表示整合。 |
| **P2** | 補助的・仕上がり品質・将来の回帰防止として重要なケース。例: aria、補助テキスト、console-only の扱い、細かい表示揺れ、保険的ケース。 |

---

## 最重要スモークテスト（15 ケース）

日常確認・リリース前の短時間確認用。各カテゴリをバランスよく含む。

| # | ケース | 優先度 | 理由 |
|---|--------|--------|------|
| 1.1 | 新規保存・同期成功 | P0 | ログ保存の基本フロー。ここが壊れると運用不可。 |
| 1.4 | 保存中は二重送信されない | P0 | 二重登録・データ破損を防ぐ。 |
| 2.1 | 未同期ログの自動再送 | P1 | オフライン復帰後の同期回復。 |
| 3.1 | 同期エラー行の再同期成功 | P0 | 手動復旧の主経路。 |
| 4.1 | アップロード成功 | P0 | 画像アップロードの基本。 |
| 4.2 | アップロード失敗 | P0 | 失敗時に中途半端更新しない。 |
| 5.1 | 削除成功 | P0 | 画像削除の基本。 |
| 5.2 | 削除失敗 | P0 | 失敗時に削除済みに見えない。 |
| 6.1 | 画像削除に伴う Blob 削除成功 | P1 | ストレージ整合の確認。 |
| 7.1 | スマホで画像変更 → PC で確認 | P1 | クロスデバイス正本の反映。 |
| 8.3 | ChipDetail でフォーカス復帰 | P1 | 画面復帰で詳細画像が最新になる。 |
| 9.1 | 保存ボタン連打 | P0 | 多重送信防止の代表。 |
| 9.3 | アップロード中に削除 | P1 | 画像操作の排他制御。 |
| 10.2 | 保存後の同期失敗 | P0 | 主要エラー表示の代表。 |
| 10.4 | 画像アップロード失敗 | P0 | 画像系エラー表示の代表。 |

## 1. ログ保存

| 優先度 | # | テスト観点 | 操作手順 | 期待結果 |
|--------|---|------------|----------|----------|
| P0 | 1.1 | 新規保存・同期成功 | 必須項目を入力し「保存」を押す（ネット接続あり） | Dexie に保存される。cloudSyncStatus = synced。一覧で当該ログが「同期済み」と表示される。フォームがクリアされる。 |
| P1 | 1.2 | 新規保存・オフライン | ネットワークを切断してから「保存」を押す | Dexie に保存される。cloudSyncStatus = local_only（または sync_error）。一覧で「未同期」または「同期エラー」と表示される。保存自体は成功する。 |
| P0 | 1.3 | 更新・同期成功 | 既存ログを選択し編集して「更新」を押す（ネット接続あり） | Dexie が更新される。cloudSyncStatus = synced。一覧で「同期済み」と表示される。 |
| P0 | 1.4 | 保存中は二重送信されない | 保存ボタンを連打する | 1 回だけ保存処理が走る。ボタンが「保存中...」または「更新中...」になり disabled になる。 |
| P2 | 1.5 | バリデーションエラー時 | 必須項目を空のまま、または周速モードで Vc/D 未入力で送信 | 保存処理は走らない。エラーメッセージ表示。saving が true のまま残らない（ボタンが再度押せる）。 |

## 2. ログ同期（自動・状態表示）

| 優先度 | # | テスト観点 | 操作手順 | 期待結果 |
|--------|---|------------|----------|----------|
| P1 | 2.1 | 未同期ログの自動再送 | 未同期ログがある状態で起動、またはネット復帰する | 起動時 effect で未同期ログが再送される。「未同期ログを再送中...」が一瞬表示される可能性あり。成功時は cloudSyncStatus = synced になる。 |
| P1 | 2.2 | 自動再送失敗時 | 未同期ログがある状態でネット不通のまま起動 | cloudSyncStatus = sync_error のまま。一覧で「同期エラー」と表示。alert 連打しない。 |
| P2 | 2.3 | 同期状態の表示整合 | 一覧を表示する | local_only → 「未同期」、synced → 「同期済み」、sync_error → 「同期エラー」と表示される。aria-label と文言が一致する。 |
| P1 | 2.4 | クラウドログ一覧取得失敗 | クラウドログ一覧取得が失敗する条件（オフライン等） | 「クラウドのログ一覧を取得できませんでした。あとで再試行してください。」と表示。生の Error.message は出ない。 |

## 3. 手動再同期

| 優先度 | # | テスト観点 | 操作手順 | 期待結果 |
|--------|---|------------|----------|----------|
| P0 | 3.1 | 同期エラー行の再同期成功 | 「同期エラー」のログで「再同期」を押す（ネット接続あり） | ボタンが「同期中...」になり disabled。成功後 cloudSyncStatus = synced。一覧で「同期済み」に変わる。resyncError が残らない。 |
| P1 | 3.2 | 再同期失敗 | 「再同期」を押した状態でネット失敗 | 一覧上に「同期に失敗しました。あとで再試行してください。」が表示される。当該行は「同期エラー」のまま。ボタンは再度押せる。 |
| P1 | 3.3 | 同一行の再同期連打 | 同じログの「再同期」を連打する | 2 回目以降は処理が走らない（その行のみ「同期中...」で disabled）。 |
| P2 | 3.4 | 他行の再同期 | 行 A が同期中に行 B の「再同期」を押す | 行 B の再同期も開始できる。行 A・B それぞれが独立して「同期中...」になる（syncingLogIds で行ごとに管理）。 |

## 4. 画像アップロード

| P1 | 4.4 | アップロード後の状態 | アップロード成功直後 | cloudImageUrl が string。hasCloudImage = true。削除ボタンが表示される。 |

## 6. Blob 削除

| 優先度 | # | テスト観点 | 操作手順 | 期待結果 |

## 7. PC ↔ スマホ同期（クロスデバイス）

| 優先度 | # | テスト観点 | 操作手順 | 期待結果 |
|--------|---|------------|----------|----------|
| P1 | 7.1 | スマホで画像変更 → PC で確認 | スマホでチップ画像をアップロードまたは削除し、PC で当該チップ詳細を開く（または画面復帰） | PC で get-chip-image-url 再取得が走り、最新の画像有無・URL が反映される。表示が矛盾しない。 |
